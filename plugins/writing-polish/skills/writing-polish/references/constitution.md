# xuan-jiang Constitution v5.0（按文体切片的 LLM Judge 成文宪法）

> v4.3 anti-ai-taste-anchors.md 是给 `scan-ai-taste.sh` 看的**字面 anchor SSOT**（230+ 条逐字红线）。本文件是给 **LLM Judge** 看的**场景化判例 SSOT**——同一批红线按文体切片、嵌入例外清单与 1-2 句 anchor 改写示例，让 judge prompt 只引用与当前文体相关的切片，缩短 token、提升 precision。
>
> 两份文件通过红线 ID（如 §1.4.111、§1.5.1）双向交叉引用，不重复全文。
>
> **使用方式**：`prompts/llm-judge-<genre>.md` 加载本文件对应章节 + 通用红线 + 例外清单，组装成最终 system prompt。

### 8 种文体

| ID | 文体 | 主要交付场景 |
|---|---|---|
| G1 | 规范性公文 | 部委 / 协会 / 处室对下属的通知 / 通报 / 决定 / 命令 |
| G2 | 领导讲话稿 | 内部会议 / 干部大会 / 现场致辞 |
| G3 | 调研报告 | 部门内部专题调研 / 行业研究 / 现状梳理 |
| G4 | 述职报告 | 干部年度 / 任期述职 |
| G5 | 汇报稿和发言稿 | 向上级 / 同行 / 第三方做工作汇报 |
| G6 | 随笔杂文 | 个人观点 / 思辨 / 评论 |
| G7 | 自媒体写作 | 公众号 / 知乎 / 小红书等社交平台 |
| G8 | 第三方咨询报告 | 咨询机构对甲方实施方案 / 对照启示 / 调研报告（v4.3 新增） |

### 5 个评分维度（D1-D5，咨询报告 5 维 rubric SSOT）

| 维度 | 名称 | 0-3 分判定 | 备注 |
|---|---|---|---|
| D1 | 标点 / 格式 | 0=零硬红线 / 1=1-2 处非中文标点等 / 2=多处 / 3=系统性违规 | Layer 1 已硬扫，此维度多为 corroboration |
| D2 | 显式 AI 套话 | 0=无 / 1=偶发 1-2 处 / 2=多处密集 / 3=每段必现 | "赋能""综上所述""让我们一起" |
| D3 | 隐喻强度 | 0=平实 / 1=个别 / 2=多处战斗化 / 3=全篇戏剧化叙事 | 「防火墙」「闸门」「跑通」「翻车」「三层防御」 |
| D4 | 大厂 vs 党政语境失配 | 0=语境一致 / 1=1-2 处串味 / 2=多处 / 3=全篇错位 | 「对标」党政合法 vs 大厂赛马；「颗粒度」「抓手」 |
| D5 | 整体散文 AI 体 | 0=具体克制 / 1=偶尔空洞 / 2=明显模板感 / 3=全篇假大空 | 句长方差、段首同质、僵化收尾、过度归纳导语 |

**Unknown 逃生舱**：模型对某段落 D_i 维度无法判定（如证据不足、文体歧义、语境不明）时输出 `"unknown"`，**禁止强行打分**。Unknown 段落不计入 κ 计算，但单独统计 unknown 率。

## §1 全文体通用硬红线（所有 G1-G8 均适用）

> 任一硬红线命中 = D_i 满 3 分。LLM judge 在 D1-D2 上首要识别本节。

### §1.1 标点 / 格式（→ D1）

来源：anti-ai-taste-anchors.md §1.4.101-117

### §1.2 显式 AI 套话（→ D2）

来源：anti-ai-taste-anchors.md §1.1（中文）+ §1.2（英文）+ §1.6（元注释）

**核心 50 条**：赋能 / 重塑 / 闭环 / 抓手 / 链路 / 打造 / 助力 / 深度融合 / 多维度 / 体系化 / 跨界融合 / 提质增效 / 接住 / 共情 / 看见你 / 令人印象深刻 / 不可或缺 / 至关重要 / 独树一帜 / 蓬勃发展 / 在某种意义上说 / 可谓是 / 值得注意的是 / 综上所述 / 由此可见 / 本质上 / 首先...其次...最后 / 需要指出的是 / 不可否认的是 / 业内人士指出 / 相关研究表明 / 大量实践证明 / 普遍认为 / delve / tapestry / underscore / pivotal / vibrant / showcase / foster / multifaceted / meticulous / seamless / furthermore / in conclusion / it's worth noting / game-changer / boasts / embark on / dive deeper / harness。

**元注释开头**（§1.6.1-1.6.5）：以下是几点 / 我将从 X 方面 / 让我为您整理 / 作为一个 AI 助手 / 以上仅供参考 / 希望对您有帮助 / 我们都知道 / 让我们一起。

判定要点：单 anchor 命中即扣 D2，**密集出现**（≥ 3 处/200 字）判 3 分。

### §1.3 句式结构红线（→ D5）

来源：anti-ai-taste-anchors.md §1.3.81-100 + §3 结构反模式

- 否定平行结构：`不是 X，而是 Y` / `不仅 X，更是 Y` / `与其说 X 不如说 Y`
- 三连排比：`A、B、C 三段并列形容` / `不是...不是...而是...` / `既...又...更...` / `从...到...到...`
- 模糊归因：`业内人士指出` / `相关研究表明` / `普遍认为`
- 僵化收尾段、雅化变奏（同概念多同义词替换）、回避"是"（"作为""可视为""体现为"代替）

## §2 文体专属红线 / 例外

### §2.1 G1 规范性公文

**底层逻辑**：政治性 / 规范性 / 指导性三维度 + 克服泛泛而谈 / 克服照抄照转 / 克服文牍主义三原则。

**追加红线**：
- 命令 / 通知 / 函混用（发文主体与文种不匹配）
- 商量口气模棱两可（"可能 / 也许 / 大概"作判断词）
- 缺具体抓手（仅泛泛号召，无量化指标）
- 程序员 / 产品经理腔（§1.5.4 在 G1 升为硬红线）

**保留项（NOT AI 味）**：
- 「一是 / 二是 / 三是」党政标准列举法（区别于「首先 / 其次 / 最后」三段式）
- 「（一）（二）（三）」中文层级编号
- 「业已 / 基本 / 总体 / 较为」软化判断词
- 「同时 / 此外 / 在此基础上 / 进一步」党政连接词
- 「按照…要求 / 围绕… / 聚焦… / 依托… / 基于… / 通过…」段首引入语

**例外**：政策原文引用段（如直引《十四五规划纲要》《二十大报告》原文）所有红线豁免，按引文规则保留。

### §2.2 G2 领导讲话稿

**底层逻辑**：演讲属性 + 口语化 + 现场感 + 鼓动力。

**追加红线**：
- 全篇书面语腔（无现场互动 / 无具体人事案例）
- 套词反射弧（"同志们 / 一会儿 / 在这里"开头每段都来）
- 段首高度同质化（每段都"我想说"）

**保留项**：
- 第一人称「我」「我们」适度使用（区别 G1 严格非人称）
- 现场感互动句（"在座的同志""刚才李局也讲了"）
- 「一是 / 二是」党政列举仍合法

### §2.3 G3 调研报告

**底层逻辑**：现状梳理 + 问题诊断 + 政策建议三段闭环；数据驱动。

**追加红线**：
- 模糊归因密集（"相关研究表明 / 业内人士指出"满篇）
- 缺数据 / 缺案例的判断（每段都是"显著提升""大幅增长"无数字支撑）
- 段尾"体现了 / 反映了 / 彰显了"分词短语挂总结（§3.15）

**保留项**：
- 「研究表明 / 数据显示」+ **具体来源与时间** = 合法
- 「敦煌研究院项祖怡 1985 年指出」点名式归因 = 应主动使用

### §2.4 G4 述职报告

**底层逻辑**：以"我"为主语的工作交代；客观成绩 + 主观反思。

**追加红线**：
- 自我吹嘘（"取得突破性进展""开创性工作"无证据）
- 总结过度模板化（"一年来 / 在领导下 / 在同志们配合下"全篇套话）

**保留项**：
- 「初步建立 / 显著提升 / 取得阶段性进展」软化成绩判断词合法
- 「下一步」「存在不足」必须保留段落

### §2.5 G5 汇报稿和发言稿

**底层逻辑**：信息透明 + 重点突出 + 时间精确。

**追加红线**：
- 缺时间锚点（"近期 / 最近 / 不久前"代替"2026 年 3 月 15 日"）
- 缺数据（汇报必须含统计数）

**保留项**：
- 党政标准列举 + 数据 + 时间三件套

### §2.6 G6 随笔杂文

**底层逻辑**：留犹豫 / 留偏执 / 留具体 / 留人称——anti-ai-taste 核心原则。

**追加红线**：
- 假装平衡（任何观点都"一方面…另一方面…"，§3.9）
- 升华反射弧（任何主题都收在"前景广阔 / 未来可期"，§3.10）

**保留项**：
- 第一人称「我」「我认为」「我倾向」自由使用
- 不完整的、犹豫的、偏执的判断 = 人味

### §2.7 G7 自媒体写作

**底层逻辑**：抓眼球 + 信息密度 + 个人视角。

**追加红线 — 同时也是橙线降级**：自媒体场景下 §1.5.3 网络口语降为软警告（B 站 / 小红书爆款腔在自媒体合法），但 §1.5.2 大厂黑话仍是硬红线（即使在自媒体场景，"赋能 / 抓手"也暴露 AI 味）。

**保留项**：
- 「本文」「本号」自指代合法
- emoji 视场景酌情使用

### §2.8 G8 第三方咨询报告（cicpa 治理沉淀）

**底层逻辑**：第三方咨询机构对甲方交付物 ≠ 甲方自家处室文件。详 anti-ai-taste-anchors.md §1.8。

**追加红线**（cicpa 5 条专属约束）：

| 锚 ID | 红线 | 改写要点 |
|---|---|---|
| §1.8.1 | 「为贯彻落实 X 协会规划」「按 Y 战略要求」政治承接句开篇 | 直接进入判断：现状结论 / 方法论说明 |
| §1.8.2 | 把判断埋到段尾「综上所述」「体现了 / 反映了」总结式套壳 | 结论先行 + 证据支撑，首段给判断 |
| §1.8.3 | 「推荐采用 X 公司方案」「Y 厂商解决方案最优」厂商背书 | 用功能描述代替厂商名；多家国际协会并列时给完整缩写不偏向 |
| §1.8.4 | 「其一 / 其二 / 其三」用在常规分项论证段 | 仅请示事项段使用；常规列举用「一是 / 二是」 |
| §1.8.5 | 涉跨主体收益重分配 / 跨地区数据主权 / 跨级权属变更话题展开「改革主张」 | 保持现状描述层面，不出具体改革主张（后者是甲方决策权范围） |

**保留项**：
- 「项目组按照…原则，调研…机构」直接讲调研方法
- 锚本 06（cicpa 053 治理后正向）是 G8 写作标杆

## §3 项目级例外清单（cicpa 6 类高频假阳）

> v4.3 grep scanner 在以下场景会误报，LLM judge **必须识别并豁免**。识别方法：上下文 ±2 行扩窗 + 关键词白名单。

### §3.1 政策文号 / 时间线豁免

**触发场景**：括号内法条文号、施行日期，如 `（国发〔2022〕14号）`、`（2017年6月1日施行）`、`（公网安〔2025〕1001号）`。

**判定规则**：括号内符合"〔YYYY〕NNN号"或"YYYY年M月D日施行"格式 → §1.4.103 不命中。

**例**：「《数据安全法》（2021年9月1日施行）」合法，不算补充叙事。

### §3.2 「其一 / 其二 / 其三」请示事项段豁免

**触发场景**：咨询报告 G8 末段含「拟请甲方审议」「提请协会决策」「呈请批准」时使用「**其一，**……**其二，**……**其三，**……」加粗结构。

**判定规则**：段落语义属请示 / 决策提请 → §3.6（请示稿专属符号）不命中。

**反例**：常规分项论证段用「其一 / 其二 / 其三」=> FAIL，应改为「一是 / 二是 / 三是」。

### §3.3 「国产化」「信创」概念豁免

**触发场景**：涉政治敏感概念时，引用官方文件原文表述（如《"十四五"国家信息化规划》《关于推动信创产业发展的若干意见》）。

**判定规则**：上下文 ±2 行含「《...规划》」「《...意见》」「政策原文」「文件指出」等文件引用标记 → 该概念按官方用语豁免。

**反例**：在非引文段落滥用「国产化替代」「信创闭环」=> 触发 §1.5.2 大厂黑话红线。

### §3.4 政策原文引用整段豁免

**触发场景**：报告引用《党的二十大报告》《政府工作报告》《十四五规划纲要》原文段落。

**判定规则**：引文段标记清晰（如使用引号、`>` 引用块、注明来源）→ 段落内所有 anchor 豁免。

**反例**：把官方文件原文打散重写后混入正文 => 不豁免，按 AI 套话判定。

### §3.5 千分位 / 计量单位括号豁免

**触发场景**：括号内含数据计算说明，如 `（130 人 × 100 元/人 = 1.3 万元/年）`、`（数百万元级）`、`（按需采购）`。

**判定规则**：括号内符合"数字 + 运算符 + 单位"或"数据规模描述" → §1.4.103 不命中。

### §3.6 「审议确定」「审议通过」豁免

**触发场景**：公文 / 咨询报告末段提请甲方审议，使用「呈请审议确定」「提请理事会审议通过」「待协会党组审议批准后」。

**判定规则**：上下文含「提请」「呈请」「请审定」「请批示」+「审议 / 批准 / 确认」组合 → 该短语按公文标准用语豁免。

**反例**：在正文段落滥用「审议确定」做判断词 => 触发模糊判断红线。

## §4 上下文白名单（v4.3 扩窗豁免，LLM judge 必须遵守）

来源：scan-ai-taste.sh 的 ±2 行扩窗逻辑。LLM judge 在判定下列 anchor 时同样按白名单豁免。

### §4.1 「防火墙」IT 实物语境豁免

白名单关键词（±2 行扩窗）：`机房 / 等保 / GB/T 22239 / 服务器 / 端口 / WAF / NGFW / IDS / IPS / 部署 N 台 / 网络安全 / 数据中心 / DMZ`。

命中 = IT 设备术语，**非战斗化隐喻**，不扣 D3。

反例：「我们用三层防御加多重防火墙保护核心数据」无 IT 关键词 → D3 = 3 战斗化隐喻 FAIL。

### §4.2 「对标」党政咨询语境豁免

白名单关键词（±2 行扩窗）：`政府工作报告 / 党中央 / 二十大 / 二十届 / 总书记 / 同级 / 同业 / 国际 / 一流 / 先进 / 启示 / 案例 / 经验 / 高质量发展 / 十四五 / 十五五`。

命中 = 党政标准用法 / 咨询报告对照分析，不扣 D4。

反例：「我们的产品要对标行业第一名跑赢竞品」无党政关键词 → D4 = 3 大厂黑话 FAIL。

### §4.3 「核心」长文密度豁免

文长 ≥ 1000 句时，「核心」「核心系统」「核心数据」「核心模块」属 IT 行业必备术语，软阈值放宽至 ≤ 15 处/千句。

### §4.4 合规括号 7 类豁免

§1.4.103 不检测以下 7 类括号（重申）：
1. 法条文号引用、2. 法律施行日期、3. 标准缩写首释、4. 表格数据备注、5. 图表标题内嵌、6. 计算说明、7. 章节自引。

## §5 评分校准 anchor 改写示例（few-shot 池）

> 每个示例覆盖 1-2 个维度，从 cicpa 053-WS3 / WS4 治理 commit 对照对中抽取，供 judge prompt 内嵌 few-shot 使用。

### Example A：D2 = 3 → D2 = 0（咨询报告，G8）

**Before（before-治理）**：
> 报告将充分赋能协会业务，深度融合人工智能能力，形成完整业务闭环，多维度提升服务质量。

**After（after-治理）**：
> 报告分析人工智能技术在协会会员服务、行业自律、继续教育三类业务中的可落地场景，给出分阶段实施建议。

**diff**：删除「赋能 / 深度融合 / 闭环 / 多维度」4 处大厂黑话 + 一组三段排比；改写为「分析 / 给出」具体动词 + 「会员服务、行业自律、继续教育」具体场景列举。

### Example B：D3 = 0（IT 实物语境，G3/G8）

**正向用例**（不扣分）：
> 机房按等保三级要求建设。边界部署防火墙 2 台、入侵检测设备 2 台，覆盖南北向流量管控。

**判定**：「防火墙」在 IT 设备语境（机房 / 等保 / 部署 N 台 / 流量管控）合法，D3 = 0。**注意**：本例已在 evals/fixtures/it-firewall.md，calibration set 不重用。

### Example C：D3 = 3（戏剧化隐喻，G6/G7）

**Before**：
> 我们用三层防御加多重防火墙保护核心数据，让 AI 不能跑通破坏闸门。

**After**：
> 我们采用边界控制、流量审核、访问授权三步检查，限制 AI 系统对核心数据的访问范围。

**diff**：「三层防御 / 多重防火墙 / 跑通 / 闸门」全部为隐喻战斗化叙事，无 IT 关键词扩窗豁免，D3 = 3。

### Example D：D4 = 3（党政 vs 大厂串味，G1/G3）

**Before**：
> 协会要对标行业最佳实践，找准抓手，跑赢同行，把数字化转型做出闭环。

**After**：
> 协会要参照同业经验，明确切入点，逐步推进数字化转型工作。

**diff**：「对标」无党政白名单关键词（无"先进 / 一流 / 政府工作报告"等），「抓手 / 跑赢 / 闭环」三个大厂黑话叠加 → D4 = 3 全篇串味。

### Example E：D5 = 3（整体散文 AI 体，G3）

**Before**：
> 综上所述，本调研报告全面分析了协会信息化现状，深入探讨了数字化转型的关键路径，从战略层、战术层、执行层三个层面提出了切实可行的实施建议。这些建议不仅具有理论价值，更是实践层面的指南。

**After**：
> 本报告分析协会 2024 年信息化现状，给出 6 项实施建议（含 OA 系统升级、统一身份认证、数据中台等），按 12 个月、24 个月两个时间节点分阶段落地。

**diff**：删除「综上所述 / 全面 / 深入 / 切实可行 / 不仅...更是 / 理论价值 / 实践指南」全套 AI 体；改写为「2024 年 / 6 项 / OA / 12 个月」具体数据时间。

**Before**：
> 关键：在这个数字化的时代，企业必须 leverage AI 技术，让我们一起 embark on 数字化转型的征程! 希望这些干货对您有帮助 — 有任何疑问随时找我!

**diff**：每段加粗冒号开头（§1.4.114）+ 半中半英（§1.4.113）+ 元注释客服话术（§1.6）+ ASCII em-dash（§1.4.101）+ ASCII 直引号 / 感叹号 → D1 = 3 + D2 = 3。

## §6 模糊词 / 推断词雷达（D5 辅助识别）

LLM judge 在 D5 判定时，遇到以下模糊词应**主动质询**："这段话有具体证据吗？"

- 「应当 / 可能 / 大概 / 估计 / 大致 / 大体 / 整体而言」（除公文软化判断保留项）
- 「显著 / 明显 / 大幅 / 极大 / 大大」（无数字支撑）
- 「研究表明 / 数据显示 / 业内人士指出」（无具体来源时间）
- 「相关 / 有关 / 各方 / 多方」（无具体主体）

判定要点：单段连续出现 ≥ 3 个模糊词 → D5 ≥ 2。

## §7 用法约定（给 judge prompt 起草人）

1. **不引全文，只引切片**：`prompts/llm-judge-research-report.md` 加载 §0 + §1 + §2.3（调研报告专属）+ §3 + §4 + §5 中相关 examples，**不要把整个 constitution 塞进 prompt**（token 浪费）。
2. **Few-shot 来自本文件**：§5 6 个 examples 是 calibration 之外的独立池，可直接 inline 到 judge prompt。
3. **Unknown 强制保留**：任何维度模型把握不准都输出 `"unknown"`，**禁止后处理填默认值**。
4. **当前日期注入**：judge prompt 末尾必须 inline 当前日期（如 `当前日期：2026-05-20`），防止 reviewer 把已发布文号（如〔2025〕1001号）误判为"未来文号"。
5. **项目认可格式白名单**：cicpa 6 类豁免（§3）+ 4 类上下文白名单（§4）必须 inline 到 judge prompt 的 `{{project_exemptions}}` 占位符。

## §8 维护原则

- **本文件不收新红线**：新红线先进 anti-ai-taste-anchors.md（v4.3 字面 anchor SSOT），通过 evals/fixtures 验证后再切片进本文件。
- **本文件按文体演进**：发现新文体或新例外，按 §2 / §3 / §4 格式追加章节，**禁止改写已落实的红线 ID**（破坏 prompt 引用）。
