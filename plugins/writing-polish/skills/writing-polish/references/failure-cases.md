## 案例 1：戏剧化叙事 + 大厂黑话叠加

### 原文（来自 README v3.1 草稿）

<!-- scan-skip -->
> 本仓库提出了一套三层防御工作流：第一层是写作前的预防机制，第二层是写作中的心理 grep，第三层是交付前的自动闸门。这套防御工作流装上之后，AI 翻车的概率会大幅下降。
<!-- /scan-skip -->

### scan 输出

```
✗ §1.5.1 战斗化叙事: 4 处（三层防御 / 闸门 / 装上 / 翻车）
✗ §1.5.3 网络口语: 1 处（本仓库）
```

### 原因分析

戏剧化叙事是 AI 把"产品文案 / 游戏文案"反射到正式公文的典型副作用。具体看：

- "三层防御"——把检查工作流伪军事化
- "闸门"——伪机械化
- "装上之后"——伪安装姿态
- "翻车"——B 站抖音爆款词
- "本仓库"——自媒体腔

### 改写过程

第一轮（机械替换，不通过）：

<!-- scan-skip -->
> 本仓库提出了一套三步防御工作流：第一层是写作前的预防机制，第二层是写作中的心理 grep，第三层是交付前的自动检查。这套防御工作流配置之后，AI 出错的概率会下降。
<!-- /scan-skip -->

机械改"三层"为"三步"，但"防御"本身仍是战斗化叙事；"配置"代替"装上"也未根本改善。scan 仍 FAIL。

第二轮（重构，通过）：

> 这个项目采用三步检查机制：写作前必读规则与范例、写作中做心理审查、交付前跑自动扫描脚本。三步配齐之后，输出文稿不达标的比例显著下降。

### 学到的

- 改不动时跳出"找近义词"思维，**改成"事实陈述句"**。原文是描述工作流，改写后就老实描述工作流。
- 把"本仓库"改成"这个项目"或"本指南"，也可以直接去掉主语。

## 案例 2：模糊归因 + 数据缺失

### 原文（某调研报告草稿）

<!-- scan-skip -->
> 业内人士指出，我国低空经济的发展正面临诸多挑战。相关研究表明，市场规模在不断扩大，但监管框架尚未健全。普遍认为，未来三到五年是关键窗口期。
<!-- /scan-skip -->

### scan 输出

```
✗ 中文红线词命中: 3 处（业内人士指出 / 相关研究表明 / 普遍认为）
✗ §3.13 问题—原因—对策机械三段论
```

### 原因分析

三句话三个模糊归因，全是"显得有依据"的反射弧。AI 没有真实信源时，倾向用"业内人士""相关研究""普遍认为"占位。详见 [citation-spec.md](./citation-spec.md)。

### 改写

> 国家信息中心 2024 年低空经济报告显示，我国低空经济市场规模 2024 年突破 6000 亿元，年增速保持在 28% 以上。但同期适航审定积压、空域分级管理细则缺位等问题仍未解决，2025 年至 2027 年是政策密集出台期。

### 学到的

- 模糊归因要么改成具体点名（"国家信息中心 2024 年报告"），要么改成承认是判断（"我们认为"）。
- 没有真实数据就先去查，不要"业内人士指出"硬扛。

## 案例 3：frontmatter 列举禁用词被 scan 误报

### 原文（SKILL.md v4.1 frontmatter description）

<!-- scan-skip -->
> ... bans em-dashes, parenthetical asides, 接住/共情/看见你, 赋能/重塑/闭环 jargon, "不是X而是Y" patterns ...
<!-- /scan-skip -->

### scan 输出

```
✗ 破折号 1 处（— 在 description 内）
✗ ASCII 直引号 1 处（"不是X而是Y"）
✗ 否定平行结构 1 处
✗ 中文红线词 7 处
```

### 原因分析

frontmatter description 必须列举禁用词，让 Claude 知道触发条件。但 description 自身也用了 em-dash 与 ASCII 直引号。这是元论述自身违反元规则的尴尬。

### 改写

把 description 改成"列举禁用词但用空格分隔，不用引号包围"：

> ... bans em-dashes and ASCII straight quotes, requires GB/T 15834 curly quotes, blocks 接住 共情 看见你 客服腔, blocks 赋能 重塑 闭环 抓手 公文黑话 ...

并把 frontmatter description 视为"规则定义元论述"，与 §4.2 心理 grep 同性质，加入 evals/README.md 的 scan 豁免边界。

### 学到的

- 元论述场景列举禁用词要用空格而非标点分隔，避免触发"否定平行""ASCII 直引号"等规则。
- 元论述与正文的边界要在 evals/README.md 明确。

## 案例 4：客服腔段尾

### 原文（某述职报告草稿结尾）

<!-- scan-skip -->
> 以上就是我对这一年工作的回顾与思考。希望对您有帮助。如有不当之处请见谅。如有其他问题，欢迎继续提问。
<!-- /scan-skip -->

### scan 输出

```
✗ §1.6.4 服务话术段尾: 3 处（希望对您有帮助 / 如有不当之处请见谅 / 如有其他问题欢迎继续提问）
```

### 原因分析

述职报告是单向陈述，不是对话。整段都是 ChatGPT 对话客服段尾贴上来的。

### 改写

> 一年回顾到此。明年的工作重点已在第四节列出，请部领导审阅指正。

### 学到的

- 正式文稿不带 ChatGPT 客服段尾，连"请审阅指正"也要简短。
- "希望对您有帮助" 严禁出现。

## 案例 5：标题层级与水平线滥用

### 原文（某 markdown 输出）

<!-- scan-skip -->
```markdown
# 工作总结

---

### 一、政治学习

---

### 二、业务工作

---
```
<!-- /scan-skip -->

### scan 输出

```
⚠ §1.7.6 跳级标题：H1 直接到 H3（跳过 H2）
⚠ §1.7.7 Thematic breaks 4 处（阈值 ≤ 5，但密度偏高）
```

### 原因分析

每个标题前都加 `---` 是 ChatGPT 输出习惯。同时 H1 直接到 H3 跳过 H2，是 layout 反射弧。

### 改写

```markdown
# 工作总结

## 一、政治学习

## 二、业务工作
```

去掉所有水平线，把 H3 升为 H2。

### 学到的

- 水平线只在文档主要分章处保留（如"附录"前），通常每篇不超过 2 至 3 处。
- 标题层级连续，H1 → H2 → H3 不跳级。

---

## 通用归纳

跑了一段时间 scan 后归纳出的反复出现 case：

1. **戏剧化词替换**：第一反应换近义词往往不通过，要回到事实陈述
2. **模糊归因**：要么具体点名，要么承认判断
3. **元论述边界**：列举禁用词的段落用 scan-skip 包围
4. **客服段尾**：删掉就行，不要"礼貌过渡"
5. **layout 反射**：水平线与跳级标题是 AI 输出指纹，整理 markdown 时一并清理
