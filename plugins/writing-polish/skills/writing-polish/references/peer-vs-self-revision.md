# 改自己 vs 改他人的辨证法

> **load-when**: Polish Protocol step 3 修改阶段（主对话提醒自己 tone 约束）；`agents/writing-reviewer.md` 子代理已内置“他批”礼貌（本文件是其依据 SSOT）。
> **automation-level**: `claude-code-session-only`（主对话 + reviewer 子代理必读）
> **来源**: 《怎样写作》（任仲然）第十二讲“修改文稿文章经验谈”
> **设计原因**: v5.x SKILL 输出独白式审稿报告，不区分自批与他批的语气差异，常出现评判作者本人的表述，伤害用户使用意愿。v6.0 补这一维。

## §1 改自己稿（解决“自己看不出毛病”）

任仲然原话：“写完即看，看不出毛病；隔几日再看，全是毛病。”

写完一稿时大脑还沉浸在“我刚写下这些”的回响里，看到的是**自己想表达的意思**，不是**纸面真实写出的内容**。这是改自己稿最大的陷阱。

### §1.1 冷读：放 24 小时再改

最简单也最有效。写完不立刻改。哪怕急稿，也强制隔 2 小时（午饭 / 散步 / 跑一个 sprint）再回来。

24 小时后看自己写的，会发现：

- 至少 3 处“我以为说清了”其实没说清的句子
- 至少 1 处“我以为论据充足”其实跳跃的论证
- 至少 5 处可以直接删掉的赘语

这就是冷读的价值。**热改稿指改自己想象中的稿；冷改稿指改纸面真实的稿**。

### §1.2 距离感的 3 种技巧

如果时间真不允许 24 h，用物理 / 感官距离替代：

| 技巧 | 怎么做 | 为什么 |
|---|---|---|
| **朗读** | 把全文从头读出来（不能默读） | 耳朵能发现眼睛看不到的拗口处和漏字 |
| **换设备** | 在 Mac 写的稿，在手机上读一遍 | 屏幕大小变化让眼睛重新“看见”内容 |
| **打印** | 打成纸读 | 物理形态让大脑切换“作者”到“读者”模式 |

任意一种至少跑一遍。**口耳并用 > 默读 3 次**。

### §1.3 不护短：把“我特别想保留的句子”挑出来重审

最难的修改不是删差句，是**删自己最得意的句子**。往往那句对自己有意义，对读者没意义。

判别标准：

- 删掉这句，整段意思是否完整？到 是到删
- 这句的作用是论据 / 推进 / 情绪渲染？到 推进 / 情绪到大概率应删
- 这句是不是“我想露一手”？到 是到一定要删

任仲然原话：“以意役法，不要以法害意。”。为了好句子破坏了主线，就是以法害意。

### §1.4 自批 tone：对自己狠

改自己稿的语气**没有**“礼貌”问题。可以毫不留情。但要区分两件事：

| 评论对象 | 怎么做 |
|---|---|
| **稿子的具体问题** | 直说：“§2 第 3 句逻辑跳跃，论据缺失” |
| **作为写作者的我** | 不评判：“我水平不行 / 我永远写不好”；禁止（无建设性） |

对稿狠，对人柔。否则会陷入“我不会写”的自我否定循环，停笔。

## §2 改他人稿（解决“批评变伤害”）

写自己的稿可以狠，改别人的稿必须柔。任仲然原话：“改别人的文章，要先想着帮人，不是想着比人高明。”

### §2.1 先复述对方意图，再指问题位置

**反模式**：

```
“这段写得不行，逻辑乱。”
```

。让作者觉得被否定，且不知道怎么改。

**正模式**：

```
“§2 这段我理解你想说的是“A 现象的 3 个原因”。你列了 X / Y / Z 三个，
但 Y 和 Z 之间有一个隐含的因果链没说明，读者可能跟不上。建议在 Y 之后
加一句“而 Y 进一步导致 ...”作过渡。”
```

。先 acknowledge 意图，再 point to 具体位置，再 give 具体改法。3 步缺一不可。

### §2.2 外科手术：具体到句，不泛评全文

**反模式**：“通篇 AI 味浓重 / 整体水平有限 / 结构松散”。大而无当。

**正模式**：“§1 段 5 句出现 3 处大厂黑话（赋能 / 闭环 / 跑通）；§3 段 2 处 em-dash；§4 末段立意散乱（请回 D5 检查）”。这种表述可定位、可修改。

**反模式比正模式糟糕的核心**：泛评让作者不知道动哪儿、动多少；外科手术让作者直接动手。

### §2.3 解释 why，不只说 wrong

<!-- scan-skip -->
```
✗ “这里不行”               （只说 wrong）
✓ “这里违反了 D2 朴实原则。
   党政文体禁戏剧化叙事，
   建议改为中性表达”        （说 wrong 也说 why）
```
<!-- /scan-skip -->

解释 why 让作者**学会**判断，不只是**被告知**判断。下次他可以自己改。

### §2.4 他批 tone：不给人设标签

<!-- scan-skip -->
| 禁忌表述（评判人） | 替代表述（评判稿） |
|---|---|
| “作者很有水平” | “本段立意清晰，结构完整”（评稿） |
| “作者写得很烂” | “§N 段 M 句存在 X 问题” |
| “这种水平也敢交付？” | “建议红线问题修复后交付” |
| “这是 AI 写的吧？” | “本段触发 D2 黑词 3 处，建议替换” |
| “你怎么想的？” | “本句的逻辑能再展开一下吗？” |
<!-- /scan-skip -->

**核心**：reviewer 评的是 **draft**，不是 **作者**。给人设标签 等于越权且无建设性。

### §2.5 何时该说“重写”

绝大多数情况建议改不建议重写。但以下两种情况可以建议重写：

1. **立意根本错了**：写的不是用户要的那回事
2. **结构完全松散**：8 段无主线，改任一段都治不了根

此时表述：

```
“本稿当前是 X 立意，但根据您的需求描述应该是 Y 立意。
建议保留 §3 §5 的具体素材（这两段有料），重新组织主线。
我可以给一个 5 段骨架草案供您参考。”
```

。仍然以“帮”的姿态出现，给具体下一步动作。

## §3 reviewer 子代理必读

v7.0 Polish Protocol step 2 触发审校时，主对话用 Task 工具 spawn [`../../agents/writing-reviewer.md`](../../agents/writing-reviewer.md)（clean context，返回自然语言反馈 + verdict，不打数值分）。

每个 reviewer 必守本文件 §2“他批”礼貌。

### §3.1 reviewer 反馈 tone 自检表（每次返回前对照）

<!-- scan-skip -->
| 自检项 | ✓ 通过 | ✗ 重写 |
|---|---|---|
| `<feedback>` 描述的是 draft 还是作者？ | draft | 作者，本条须重写 |
| 是否定位到具体段/句？ | 是 | 否，本条须重写 |
| 是否给具体改法？ | 是 | 只说“建议改”，本条须重写 |
| 整体 tone 是建议还是评判？ | 建议 | 评判，本条须重写 |
<!-- /scan-skip -->

### §3.2 典型反例到正例

| ✗ reviewer 反例 | ✓ reviewer 正例 |
|---|---|
| “通篇 AI 味浓重，结构松散” | “【AI味/标点】§2-§4 三段大厂黑话密度过高（黑词 5 处/千字），朴实原则受损；【结构】§3 段间过渡缺失” |
| “整体水平低” | “§2 句 3 出现‘赋能’；§3 句 1 出现‘闭环’；§4 段缺主题句” |
| “重写” | “将‘赋能’改为‘支撑’；将‘闭环’改为‘完整流程’；§4 加一句总起” |

### §3.3 reviewer 与作者的关系

reviewer 不是法官，是**同行**。同样会犯类似错误，同样需要被指正。这种 peer 心态让 reviewer 的建议更容易被作者接受。

任仲然原话：“以共勉之心改人之稿。”
