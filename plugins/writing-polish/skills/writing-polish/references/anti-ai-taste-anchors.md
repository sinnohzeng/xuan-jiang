## §0 核心机制（理解原理而非死记词表）

AI 味不是单点 bug，是**五层叠加**：

1. **训练目标**：RLHF 把模型推向高通过率，温和完整安全的"标准答案机"
2. **解码策略**：生成时 token 概率最高 = 统计上最像好答案的句子，不是某个真人会写的句子
3. **训练数据同质化**：互联网被 AI 反向污染，模型学到上一代模型已加工过的模板
4. **统计性思维替代语言思维**：人是先有想法再找词；AI 是预测下一个最可能的词，倾向最安全最高频
5. **戏剧化偏好（v4.1 新增）**：AI 受互联网产品文案、游戏文案、自媒体爆款标题污染，倾向用动作化、战斗化、夸张化词汇制造冲击感（如"防御""闸门""装备""跑通""翻车""引擎""赋能""链路""装上"），但正式中文书面语应平实克制。这一层是最隐蔽的 AI 味，因为 AI 自己以为是在"写得有冲击力"，实际暴露了网感

> **v6.0 落地**：硬规则解决不了的 6 类盲点（同形异义 / 新词演化 / 整体散文 AI 体 / 粗颗粒度判定 / 无 calibration / 同形近义变体逃逸）由主对话 L2 D1-D5 self-judge + L3 多 reviewer 接管。L1 regex 仍是 fast path（< 1 秒、透明可审计、零模型成本）；L2/L3 是 slow path，主对话调用方决定模型路由（参见 SKILL.md §2.2 Polish Protocol）。

**真正"去 AI 味"的核心不是改词，是把表达从「标准答案」拉回「具体表达」**：

- 留**犹豫**——不强求圆满
- 留**偏执**——不假装平衡
- 留**具体**——不滑向抽象
- 留**人称**（我、他、张三）——不藏在"研究表明"后面
- **长短句交错**，允许局部失衡
- 该用**数字**时用数字（"60%" 而不是"大幅提升"）
- 该**点名**时点名（"敦煌研究院项祖怡 1985 年" 而不是"有学者"）

**为什么是 AI 味**：RLHF 训练让 AI 在"做出判断"前加这些限定词以"显得严谨"，但密集出现就成了模板。

#### 模糊归因（6 条）— Wikipedia AIWEASEL

97. **大量实践证明**（不给案例）

### §1.4 标点 / 格式红线（17 条，含 v4.2 新增 4 条）

101. **— 破折号**（中文用户硬点，英文 em-dash 同样是 Reddit/Wikipedia 头号指纹）
102. **—— 中文长破折号**（用户硬点）
103. **括号内插入式补充**（用户硬点：xxx（即 yyy）、xxx（如 a、b、c）、xxx（也就是说...））

   **合规括号 7 类白名单**（v4.3 新增 docs，scan 不再扩展检测，仅作认知层指引——避免 AI 写作时矫枉过正回避所有括号）：

   1. **法条文号引用**：（国发〔2022〕14号）、（公网安〔2025〕1001号）—— 党政公文标准格式
   2. **法律施行日期**：（2017年6月1日施行，2025年10月28日修订通过）—— 公文必备
   3. **标准缩写首释**：（OA）、（CMS）、（AICPA）、（FRC）、（ISCA）、（PMO）、（BYOK）、（WAF）—— 第一次出现的术语缩写，符合 GB/T 15834
   4. **表格数据备注**：表格内 `(数百万元级)`、`(按需采购)` 等简短数据描述
   5. **图表标题内嵌**：`![图 X 推荐方案（方案 B）部署架构]` —— 图表注解
   6. **计算说明**：（130 人约 1.3 万元/年）—— 数据计算关系
   7. **章节自引**：（详见 §X.Y）—— 文档内部交叉引用

   **判别规则**：scan 第 134 行只检测 `（如|（即|（也就是说`三种 AI 起手式叙事补充，上述 7 类合规括号本来就不会触发。读者本能上要识别的不是括号本身，而是「括号里塞补充叙事」这一动作。
104. **冒号 + 三段分号工整列举**（X 体现在三个层面：一是...；二是...；三是...，搜狐直接命名为标点指纹）
105. **正文里残留 Markdown**（**加粗** ## 标题在纯文本场景出现）
106. **每段都有粗体强调短语**（Wikipedia 报告的 ChatGPT 主笔指纹）
107. **emoji 出现在严肃文本**（🚀📘 等装饰性 emoji 在公文 / 报告 / 学术稿中）
108. **过度引号**（文本考古学式、历史-语言双轨互证 这种 AI 自造组合词加引号包装）
109. **句号后立刻换行做金句感**（短句堆叠制造仪式感的 self-help 风）
110. **章节标题套路**：X：Y 双标题、Title Case 异常（英文每词首字母大写）
111. **中文引号必须用大陆国标弯引号 " " 和 ' '**（★用户硬点，依据 GB/T 15834-2011 § 4.8）：大陆国家标准规定中文引号是 `"` `"`（U+201C / U+201D）和 `'` `'`（U+2018 / U+2019），双层嵌套时外层双引号、内层单引号。**严禁**：(1) 中文上下文用 ASCII 直引号 `"` `'`；(2) 中文上下文用直角引号 `「」` `『』`（这是港台 / 日式标点，大陆党政公文不用）。AI 默认输出 ASCII 直引号或乱用直角引号，是中英语料混训留下的指纹。唯一例外：纯英文短语 / 代码 / 技术术语保留 ASCII 直引号。**正确**：他说"很重要"。**错误**：他说"很重要"或他说「很重要」。

112. **数学符号代替自然语言**（★用户硬点：加号 + 和等号 = 在正式中文里禁用）：AI 倾向用 `+` 表示并列、用 `=` 表示等同或定义、用 `→` 表示因果或流程、用 `&` 表示并联，是程序员 / 产品经理 / PPT 腔的指纹。这些符号在数学表达式、化学反应式、合作署名（"周杰伦+方文山"等明确并列实体）等少数场景合法，正式中文叙述里禁用。**改写要求**：**严禁机械替换**为"和""与""即"，应当**重构整段表述**回到完整中文句式。
    - 例 1（错误）：`数据 + 算法 + 算力` → 机械改成 `数据和算法和算力`（仍生硬）→ 正确改写：`数据、算法和算力构成三大要素`
    - 例 2（错误）：`代码 = 文档` → 机械改成 `代码即文档` → 正确改写：`代码本身就是文档，不再单独维护说明`
    - 例 3（错误）：`效率 + 质量` → 机械改成 `效率和质量` → 正确改写：`既要追求效率，也要保证质量`
    - 涉及符号：`+`、`=`、`→`、`⇒`、`&`、`/`（在并列短语里如"标点 / 用词" / 替换思路：列出三组逐一论述或合并为正常顿号）

113. **半中半英术语（中英文混搭）**（★用户硬点，依据 RightCapitalHQ/chinese-style-guide）：正式中文书面语禁止 `操作 checklist`、`前端 dev`、`后端 framework`、`AI workflow`、`SKILL 化训练`、`prompt 工程`、`token 化` 等中英缝合词。AI 倾向直接保留英文术语方便自己理解，但读者读起来像未翻译的草稿。**改写规则**：
    - 已有标准译名的术语必须用中文：checklist → 清单 / 检查表；workflow → 流程；framework → 框架；prompt → 提示词
    - 没有标准译名的纯技术术语首次出现可加括注：`提示词工程（Prompt Engineering）`，后续单独使用中文
    - 商品名 / 产品名 / 公司名保留英文：GitHub / Anthropic / DOCX
    - 单字母变量 / 缩写首次出现要解释：`通用唯一识别码（UUID）`

114. **每段加粗冒号开头**（v4.2 新增）：`**重点：**xxx`、`**关键：**xxx`、`**注意：**xxx`、`**核心要点：**xxx` 这类粘贴式标题。**为什么是 AI 味**：ChatGPT 输出 layout 反射弧，把任意短语强行升级为标题。**改写**：删去加粗冒号，让句子直接陈述事实，或者改用"一是 / 二是"分项列举。

115. **数字 list 1. 2. 3. 滥用**（v4.2 新增）：用阿拉伯数字加点的编号列表替代段落叙述，把 5 - 15 个 bullet 当成回答。**为什么是 AI 味**：训练数据中"列点回答"被强化，正式公文极少用。**改写**：党政公文用"（一）（二）"或"一是 / 二是"中文层级，散文用段落式叙述。例外：操作清单 / 检查表合法。

116. **标题化偏好（每段必加 ## 子标题）**（v4.2 新增）：任何稍长内容都加 H2 / H3 子标题，把每段都标题化，是 Markdown 训练数据污染。**为什么是 AI 味**：人类写作主要是流式段落叙述，党政公文一级标题之下就是连续段落。**改写**：删去多余子标题，让段落首句承担"导引"功能。每个一级标题之下不超过 1 至 2 个子标题。

117. **英文标点穿插中文段落**（v4.2 新增）：中文段落里残留英文逗号 `,` 句号 `.` 冒号 `:` 分号 `;`。与 ASCII 直引号同源，是中英语料混训残留。**为什么是 AI 味**：典型 AI 输出指纹。**改写**：换成中文标点 `，` `。` `：` `；`。例外：URL 内、代码块内、英文专名后。

## §1.6 元注释 / 客服话术红线（v4.2 新增）

> **底层逻辑**：RLHF 安全训练让 AI 在回答前后加大量元注释、自我介绍、免责声明、服务话术段尾。这些是确凿的 AI 客服腔指纹，正式书面语场景必须全清。
>
> **改写原则**：直接进入正文，不做"自我交代"。读者读的是内容本身，不是 AI 的"礼貌姿态"。

### §1.6.1 元注释开头（5 条）

正式书面语禁用：

- **以下是几点说明** / 以下是几点想法 / 下面给出
- **我将从 X 个方面来回答** / 我将围绕 X 个维度展开
- **让我为您整理一下** / 让我帮您梳理 / 让我先来分析
- **请允许我** / 请容我先 / 在此我想
- **接下来我会** / 我接下来要 / 下一步我将

**改写**：删除元注释开头，直接进入正文。例：把"以下是几点说明"删除，让正文第一句直接陈述事实或判断。

### §1.6.2 自我介绍 / 身份声明（4 条）

正式书面语严禁：

- **作为一个 AI 助手** / 作为一个 AI 模型 / 作为一个语言模型
- **作为大语言模型** / 作为对话式 AI
- **我虽然是 AI，但** / 我作为 AI 的局限性是 / 我的知识截止日期是
- **我没有 X 能力** / 我无法 X / 我不能为您 X（除非用户主动询问能力边界）

**为什么是 AI 味**：所有正式中文文档都不会有这种身份说明开头。这是 ChatGPT 客服腔训练后遗症。

### §1.6.3 免责声明 / 安全帽（5 条）

正式书面语禁用：

- **以上信息仅供参考** / 仅供参考请以官方为准 / 信息可能有偏差
- **具体情况建议咨询专业人士** / 建议咨询医生 / 建议咨询律师 / 建议咨询专业机构
- **我不能提供 X 建议，但可以** / 我无法替代专业建议
- **请注意** / 请知悉 / 需要提醒的是（孤立成段，无具体内容）
- **以上内容如有错误请指正** / 如有不当之处请见谅

**为什么是 AI 味**：RLHF 反射弧。改写：删除免责声明，事实直陈，由读者自行判断。

### §1.6.4 服务话术段尾（5 条）

正式书面语严禁：

- **希望对您有帮助** / 希望这能帮到您 / 希望这些内容对您有用
- **如有其他问题，欢迎继续提问** / 还有什么问题尽管问 / 随时找我
- **有任何疑问请随时告诉我** / 有不清楚的地方请告诉我
- **祝您 X 顺利** / 祝您工作顺利（除非确实是结尾问候场合）
- **谢谢您的提问** / 谢谢您的信任 / 感谢您的耐心

**为什么是 AI 味**：把对话客服话术贴到正式文稿尾部，是 ChatGPT 主笔指纹。改写：直接以判断或事实段收尾。

### §1.6.5 拟人化集体代词滥用（4 条）

正式书面语谨慎使用，正式公文 / 学术论文严禁：

- **我们都知道** / 我们大家都清楚 / 我们应当
- **大家都觉得** / 大家普遍认为（与模糊归因 §1.3.100 同源）
- **让我们一起来** / 让我们一起 / 让我们共同
- **不妨想象一下** / 不妨设想 / 试想一下

**为什么是 AI 味**：GPTZero 标记为高置信度 AI 指纹。读者并未与作者达成"我们"共识，强行拉读者下水。改写：用"笔者""本文""调研发现"或直接陈述客观事实。

## §1.7 Wikipedia 长尾盲区（v4.2 新增）

> 来源：Wikipedia: Signs of AI writing 完整版，含 v4.1 未覆盖的八类。这是确凿 AI 指纹，单条命中即应判 FAIL。

### §1.7.1 Reference markup bugs（ChatGPT / Grok 输出残留代码）

正式文档严禁出现以下字符串，是确凿的 AI 工具输出指纹：

- `:contentReference[oaicite:0]{index=0}` / `oaicite` / `contentReference`
- `oai_citation` / `oai_citation:0`
- `+1`（紧跟在引用后做"赞同"标记）
- `attached_file` / `attached_file:`
- `grok_card` / `grok-card`
- `[\d+]` 紧贴中文末尾（机械引用编号）

**改写**：清理所有此类标记，要么改成正常引用脚注，要么去除。

### §1.7.2 Placeholder dates（占位符日期）

严禁出现：

- `2025-xx-xx` / `2026-xx-xx` / `XXXX-XX-XX`
- `{{access-date|2025-xx-xx}}` / `{{date|YYYY}}`
- `[date]` / `[year]` / `[month]`

**为什么是 AI 味**：这是 ChatGPT 模板未填充的占位符。**改写**：要么填实际日期，要么删除该字段。

### §1.7.3 Elegant variation（雅化变奏，同义词替换）

同一概念在相邻段落用多个同义词指代，制造"行文丰富"的假象：

- 一段写"研究人员"、下段写"学者"、再下段写"专家"、最后写"业内人士"，实际指同一群人
- 一段写"项目"、下段写"工程"、再下段写"举措"、最后写"行动"，实际指同一件事

**为什么是 AI 味**：generative model 的 repetition penalty 机制副作用，AI 自动避免重复词，用同义词替换造成"主语漂移"。**改写**：同一概念固定用一个词，重复才是常态。

### §1.7.4 Avoidance of basic copulatives（回避系动词）

英文 AI 回避 is / are，中文 AI 对应回避"是"，过度用"作为""体现为""可视为""属于一种"。

**反例对照**：

- AI 写法：`数字化转型作为新阶段的一项重要工作` 或 `数字化转型可视为生产力变革的载体`
- 改写：`数字化转型是新阶段的重要工作` 或 `数字化转型推动生产力变革`

### §1.7.5 Inline-header vertical lists（粘贴式段首标题）

段首加 `Background:` `Note:` `Summary:` `Context:` `背景：` `说明：` `小结：` 等粘贴式标题，把段落变成卡片。

**改写**：删除段首贴标题，让首句承担"导引"功能。例外：技术文档中明确的"前提条件 / 输入 / 输出"模板合法。

### §1.7.6 Skipping heading levels（跳级标题）

直接从 H1 跳到 H3，跳过 H2。或者全文 H1 + H4 混用。是 AI 输出 layout 反射弧。

**改写**：层级连续，H1 → H2 → H3 不跳级。

### §1.7.7 Thematic breaks before headings（标题前水平线）

每个标题前都加 `---` 水平分隔线，是 ChatGPT 输出习惯。

**改写**：删除多余水平线，只在文档主要分章处保留，且通常每篇不超过 2 至 3 处。

### §1.7.8 "Concrete" 滥用（英文）

英文 AI 反复用 concrete / tangible 强调 "concrete evidence" "tangible results" "concrete examples"。

**改写**：删除 concrete / tangible，给具体事实数据。例：把 `concrete evidence shows` 改成 `the 2024 audit shows`。

## §1.8 咨询报告专属约束（v4.3 新增，第三方咨询机构对甲方交付物专用）

> **底层逻辑**：第三方咨询机构对甲方的实施方案、调研报告、对照启示等交付物，与甲方内设处室文件 / 部委规划 / 政府工作报告 在身份、口吻、收束三方面**根本不同**。本节 5 条专属约束帮助咨询报告作者识别身份边界，对应锚本 06（cicpa 治理后正向）与锚本 09（甲方规划反向参照）的对照。

### §1.8.1 第三方咨询机构身份边界（不锚甲方自家规划）

**禁止**：「为贯彻落实 X 协会规划」「按 Y 协会信息化战略要求」「持续以 X 规划为龙头」等政治承接句开篇——这是甲方处室文件标准格式，第三方咨询机构使用即身份越界。

**正确**：直接进入判断。开篇给出现状结论或方法论说明。锚本 06 cicpa 053 治理后开篇即「项目组按照"同级别、同规模、同属性"原则，调研五类与协会在组织性质、人员规模、业务形态具有可比性的机构」——直接讲调研方法。

### §1.8.2 结论先行 + 证据支撑

每节首段给出判断（"信息系统部署方式已由自建机房转向政务云"），后续段落用具体单位、平台、数据展开论证。读者扫读首句即可掌握主要观点。

**禁止**：把判断埋到段尾"综上所述""体现了 / 反映了 / 彰显了"——这是 AI 写作的总结式套壳。

### §1.8.3 不为特定厂商背书

用功能描述（"一体化智能平台"），不指名厂商（不写"钉钉"或"飞书"）；多家国际协会并列时给完整缩写（AICPA / FRC / ISCA / JICPA），不偏向任何一家。

**禁止**：「推荐采用 X 公司方案」「Y 厂商提供的解决方案最优」——这是销售话术不是咨询交付。

### §1.8.4 「其一 / 其二 / 其三」分级使用

**仅用于请示事项段**——咨询交付的最末段如确需提请甲方决策、审议、批准、确认时，使用「**其一，**……其二，**其三，**」加粗结构。

**非请示事项段** 用「一是 / 二是 / 三是」党政标准列举法（详见 §5）。

**禁止**：把「其一 / 其二 / 其三」用在常规分项论证段——这是请示稿专属符号，不是普通列举。

### §1.8.5 多方利益话题先静默

咨询报告中触动多方内容主权或费用流的话题（如 cicpa 项目继续教育在线平台涉北京 / 上海 / 厦门三家国家会计学院 + 地方注协费用流），不引入"AI 能力扩展""统一治理"等敏感主张，写到现状描述即止。

**判别规则**：如果某话题展开会涉及「跨主体收益重分配 / 跨地区数据主权 / 跨级权属变更」，咨询报告应保持现状描述层面，不出"具体改革主张"——后者是甲方内部决策权范围。

## §3 结构级反模式（17 条，最难洗）

> 来源：Wikipedia "Outline-like conclusions" + 搜狐"结构模板化" + 80aj"段落同质化"。这层最难自动检测，需要 Claude 主动反思。

1. **僵化收尾段** —"综上所述" / "面对未来" / "在...的道路上" 强行升华
2. **"挑战与展望"双段套壳** —"尽管面临...挑战，但..."（Wikipedia AIFUTURE 命中）
3. **"意义深远"型结尾** — 把任何主题硬升华到"具有深远意义"
4. **每段首句高度同质** — 每段都是"从...角度看"、"值得思考的是"、"更深层次..."
5. **段落顺序可换不影响理解**（搜狐"打乱测试"）— 段落间没有真正逻辑递进
6. **过度归纳的导语** — 文章开头一段"本文将从 A、B、C 三方面展开"
7. **每段必有总结句** — 人类写作不每段都收束
8. **小标题三段对仗** —"理论篇 / 实践篇 / 展望篇"、"是什么 / 为什么 / 怎么办"机械三段
9. **过分平衡的论证** — 任何观点都"一方面...另一方面..."、"既...又..."，没有真观点
10. **正能量结尾反射弧** — 任何主题都收在"充满希望" / "前景广阔" / "未来可期"
11. **过度归因到"重要性"** — 把每个事实都"highlights/underscores the importance of..."（Wikipedia "Superficial analyses"）
12. **结尾段套用模板** —"In a world where..." / "在这个 X 的时代" 开始的拔高段
13. **"问题—原因—对策"机械三段论**（公文味 + AI 味叠加最严重场景）
14. **每个论点配工整三例**（rule of three 滥用）
15. **段尾分词短语挂总结**（"...，体现了 X / 反映了 Y / 彰显了 Z" — Wikipedia 直接点名）
16. **话题漂移 / 段落主题渐变**（v4.2 新增）— 一篇文章从 X 主题逐段演变到 Y 主题，实际原本没打算写 Y。AI 缺 coherence 锁，越写越漂。检查方法：把每段首句抽出来排成一行，应该是一条递进逻辑链；如果不能合并成一句话，说明话题已漂。
17. **形容词堆砌**（v4.2 新增）— 连用 3 个以上同义形容词，如"深入的、全面的、系统的分析""扎实的、有效的、可持续的工作"。AI rule of three 的退化形态。改写：留一个最具体的形容词，删去另外两个。

### 软阈值（密度限制，v4.3 按文长动态化）

v4.3 起 scan-ai-taste.sh 自动按句子数动态调整阈值，**不再人工记忆固定阈值**。直接跑 scan 看 ✓ / ⚠ 即可：

```bash

```

阈值规则（按千句密度 ≤ base，base 默认 3 / 3 / 2 对应「已经 / 核心 / 这一」）：

| 文长（句数） | 已经 | 核心 | 这一 |
|---|---|---|---|
| < 200 句（短文） | ≤ 3 | ≤ 3 | ≤ 2 |
| 200-500（中文） | ≤ 6 | ≤ 6 | ≤ 4 |
| 500-1000（长文） | ≤ 9 | ≤ 9 | ≤ 6 |
| ≥ 1000（超长） | ≤ 15 | ≤ 15 | ≤ 10 |

**为什么动态化**：cicpa 053 实战发现 IT 行业必备术语（核心数据 / 核心系统 / 核心模块 / 核心功能）在千句以上长文中天然密度高，固定阈值 ≤ 3 偏严会逼出妥协改写。动态阈值符合「长文容忍少量术语高频」语料事实，不削弱红线（密度仅是软阈值 WARN，不影响 FAIL）。

### 句长方差（人写有长短）

```bash
python3 -c "
import re, statistics
text = open('$FILE').read()
sents = [len(s) for s in re.split('[。！？；]', text) if 5 < len(s) < 200]
if len(sents) > 5:
    print(f'句长标准差: {statistics.stdev(sents):.1f} (阈值 ≥ 8)')
"
```

## §5 重要保留项（NOT AI 味，应主动使用）

### "一是…二是…三是…" 党政公文标准列举法

**这不是 AI 味，是党政公文的标准结构**：

- 国办文件、财政部通知、国务院发展研究中心专家文章大量使用
- 见 [`assets/real-world-anchors/04-drc-data-economy.md`](../assets/real-world-anchors/04-drc-data-economy.md)
- **应当保留并主动使用**，区别于 AI 机械化的"首先...其次...最后"

#### 用法规范

- 一级：中文数字"一、二、三"
- 二级：`一是 / 二是` 或 `（一）/（二）`
- 末段请示事项：`其一 / 其二 / 其三`

#### 区分

- ❌ AI 味：全文所有分项都机械使用"首先 / 其次"或"第一 / 第二"，不分层级、不分场景
- ✅ 公文：分级使用、含具体数据、判断后接论据

### 八项文风指纹（正面对齐）

归纳自真实党政公文与国家智库咨询报告，写作时按此对标：

| 编号 | 指纹 | 具体表现 | 示例 |
|---|---|---|---|
| 1 | 段落起手用动宾短语或偏正名词短语 | 不用完整判断句起手 | "坚持统筹规划""推进整合""启动条件总体评估" |
| 2 | 句式以长句＋分号为主 | 一段一气呵成，分号衔接多组平行小句 | 一段 200—400 字，分号 3—5 个 |
| 3 | 动词偏好收敛 | 公文动词词库较小，反复使用同一批 | 推动／推进／加快／加强／强化／完善／健全／建立／实现／落实／统筹／着力／持续／深入／全面 |
| 4 | 判断词带程度软化 | 避免绝对化判断 | 业已／基本／总体／较为／相对／基本具备／显著提升／初步建立／取得突破性进展 |
| 5 | 结构引入语规范 | 段首常用固定引入短语 | 按照……要求、围绕……、聚焦……、依托……、基于……、通过…… |
| 6 | 连接词规范 | 不用互联网口语 | 同时、此外、在此基础上、进一步、为此、为进一步 |
| 7 | 数字规范 | 年份精确到月 | "2026 年 7 月前""2022 年底前"；层级编号用中文数字（一、（一）1．）；统计数据用阿拉伯数字 |
| 8 | 决策语气软化 | 不用强硬 | 建议／拟／呈请／审议批准／备案／部署／统筹／研究提出；避免"必须""一定要""绝不" |

**第一步：对标阅读（写作前）**

**第二步：对标写作（写作中）**
