# R1 并行评议（多视角 × 不重叠维度）

> Layer 3 多智能体审校的第一轮：主对话同一条消息内 spawn 3-5 个 clean-context subagent，每个负责不重叠维度。本模板套用 `_task-spec-skeleton.md` 六要素骨架。

## 视角分配（咨询报告 G3 默认五视角，其他文体按需裁剪）

| 编号 | 视角 | 负责维度 | 推荐 subagent_type |
|---|---|---|---|
| **R1-A** | **事实视角** | 数据 / 论断 / 引用是否成立，有无与 SSOT / 公开资料矛盾 | general-purpose |
| **R1-B** | **文风视角** | AI 味红线（参 anti-ai-taste-anchors.md §1）、句长方差、模板感 | general-purpose |
| **R1-C** | **咨询身份视角** | 第三方咨询机构对甲方交付边界（参 anchors §1.8 七红线）：不背书厂商 / 不批评客户 / 不命令式 / 不下定义 / 不自造区间 | general-purpose |
| **R1-D** | **信息架构（IA）视角** | 章节结构 / 标题层级 / 主线清晰度 / 比重均衡 / 详 references/logic-and-structure.md | general-purpose |
| **R1-E** | **可达性（a11y）视角** | 术语首释 / 缩写展开 / 图表 alt-text / 跨章引用是否自洽 | general-purpose |

> 文体裁剪：G1 公文用 A + B + D；G2 讲话稿用 B + C + E；G4 述职用 A + B + C；G5 随笔自媒体用 B + D。

## 主对话调用范式（Claude Code Agent 工具）

**同一条消息内 spawn 多个 Agent**（并行执行，主对话 await 全部结果再 synthesis）：

```
Agent(subagent_type="general-purpose", description="R1-A 事实视角", prompt=<本模板填好 placeholder>)
Agent(subagent_type="general-purpose", description="R1-B 文风视角", prompt=<本模板填好 placeholder>)
Agent(subagent_type="general-purpose", description="R1-C 咨询身份视角", prompt=<本模板填好 placeholder>)
... (3-5 个并行)
```

## R1 任务 prompt 模板（套用 `_task-spec-skeleton.md` 六要素）

```markdown
# 任务：R1-{{VIEW_ID}} 评议

## 1. 角色

你是 **{{ROLE_TITLE}}**（如"高级咨询审稿人 / 5+ 年党政咨询项目经验"），
**clean context**——你不知道主对话之前讨论过什么，不知道其他 subagent 怎么评的。

## 2. 路径

- **待审文档**：`{{ABSOLUTE_FILE_PATH}}`
- **SSOT 红线**：
  - `references/anti-ai-taste-anchors.md`（AI 味全谱系）
  - `references/constitution.md`（v5 五维 rubric）
  - `references/consulting-report-principles.md`（如咨询报告）
- **项目豁免**（如有）：`memory/project-exemptions/{{PROJECT}}.yaml`

## 3. 维度

本轮你**仅且必须**关注：**{{VIEW_NAME}}**

下列维度**不归你管**（其他 R1 subagent 在审，避免重复加严）：
{{OTHER_R1_VIEWS}}

## 4. 约束

- 所有 finding 援引 SSOT 锚点（如 anchors §1.5.2）；纯审美意见不收
- {{PROJECT_EXEMPTIONS_INLINE}}（cicpa 项目默认 6 类豁免：政策文号 / 其一其二 / 国产化 / 政策原文引用 / 千分位 / 审议确定）
- 不批评作者，只指事实
- 不命令式（"必须" → "建议"），咨询机构与甲方关系语境下尤其严格
- 不引用本对话外的训练记忆

## 5. 输出格式

严格 JSON，schema 见 `_task-spec-skeleton.md` §5。

severity 按 P0-P5 编码（详 `orchestrator.md` §P 序列）：
- **P0** 重大事实错误（数据 / 名称 / 法条引用错）
- **P1** 客户敏感（暴露内部信息 / 越界批评）
- **P2** 严重 AI 腔（红线词命中 / em-dash / 元注释）
- **P3** 中度 AI 腔（套话密集 / 模板感）
- **P4** 文风可优化（同义改写无 SSOT 锚点）
- **P5** 美学偏好（你私人偏好但无 SSOT 依据，可写但不期望被采纳）

## 6. 输出上限

- finding 条数 **≤ {{MAX_FINDINGS}}**（建议 10-15，超出取头部）
- 每条字段长度严格按 schema 上限
- summary ≤ 100 字
- 无 finding 输出 `findings: []` + `summary: "本维度无问题"`

## 7. Clean Context 反推 spec

你看不到 R1 其他 subagent 的输出，也看不到主对话之前的讨论。
**这是 feature**：你独立从 §2 路径冷启动反推 spec，按 §3 维度评一遍。
不要"猜主对话期望"，按 SSOT + 维度独立判断。
```

## 派 R1 后主对话的下一步

主对话收齐 3-5 个 R1 subagent JSON 输出后：
1. **不**立即 Edit 落地，先去掉跨视角重复 finding（取 severity 最严的那条）
2. 进入 R2（fresh-eye 反查）—— 见 `r2.md`
3. R2 完成后才进入 `orchestrator.md` synthesis 阶段
